# GitHub Copilot クレジット消費量計算ツール 計算ロジック説明

**対象読者**: 全社員向け（承認プロセス、クレジット管理、個人の事前見積もり用）  
**本ツール目的**: 1リクエスト（Chat/CLI）または1PR（Code Review）あたりのクレジット消費量を**事前に見積もる**スタンドアロン Web ツール

---

## 用語定義（5分で分かる基礎）

| 用語 | 説明 | 例 |
|---|---|---|
| **トークン** | AI モデルが文字列を処理する際の最小単位。日本語は1文字≒0.65トークン、英語は1文字≒0.25トークン | 「こんにちは」(5字) ≒ 3.25 トークン |
| **Input トークン** | AI に送信する情報（プロンプト＋参照ファイル）のトークン数 | 500 字プロンプト＋10KB資料 ＝ 約5,000トークン |
| **Output トークン** | AI が返す回答のトークン数 | 1,000字回答 ≒ 650 トークン |
| **Cached Input** | 前回リクエストで使用した参照情報を AI キャッシュから再利用した場合のトークン数（単価が通常の1/10） | 同じ資料を2回目に使用 → キャッシュ効果で安価 |
| **クレジット** | GitHub Copilot の課金単位。1クレジット = $0.01 | 5,000トークン = 0.5〜1.5クレジット（モデルにより異なる） |

---

## 全体構造(2つの計算式)

本ツールは「①Chat/CLI」と「②Code Review」で計算式が完全に別物です。

---

### ① Chat / CLI の消費クレジット（基本計算式）

```
消費クレジット = [(Input × Input単価) + (Cached Input × Cached単価) + (Output × Output単価)] ÷ 10,000
```

**登場人物:**
- **Input トークン数**: プロンプト + 参照情報（再利用されない分） + 過去ターン + オーバーヘッド
- **Cached Input トークン数**: 参照情報のうち「キャッシュから再利用できた分」
- **Output トークン数**: AI の回答文
- **モデル別単価**: Input/Cached Input/Output それぞれで異なる固定値（例: Claude 3.5 Sonnet は Input 0.3円/1000トークン）

---

#### 📊 実例ベース計算（シナリオ別）

**【Scenario A】1回限りの Chat リクエスト（参照なし）**

```
入力内容:
  - プロンプト文字数: 500字
  - 参照ファイル・テキスト: なし
  - 過去会話ターン数: 0（初回）
  - 想定回答文字数: 300字

計算ステップ:
  1. プロンプトのトークン数 = 500字 × 0.65 = 325トークン
  2. 参照情報トークン = 0（参照なし）
  3. Chat 固定オーバーヘッド = 1,500トークン
  4. Input合計 = 325 + 0 + 1,500 = 1,825トークン
  5. Cached Input = 0（参照がないため）
  6. Output = 300字 × 0.65 = 195トークン
  7. クレジット消費 = [(1,825 × 0.3) + (0 × 0.03) + (195 × 1.5)] ÷ 10,000
                  ≒ (547.5 + 0 + 292.5) ÷ 10,000 ≒ 0.084クレジット (約0.84円)
```

**【Scenario B】同じ Word 資料を使った 2 回目の Chat**

```
入力内容:
  - プロンプト文字数: 500字（1回目と同じ資料について質問）
  - Word 添付ファイル: 500KB
  - 参照情報の再利用状況: 「同じ資料で2〜3往復」= 35% キャッシュ率
  - 過去会話ターン数: 0
  - 想定回答文字数: 500字

計算ステップ:
  1. プロンプト = 500字 × 0.65 = 325トークン
  2. Word資料のトークン = 500KB × 300(KB単価) = 150,000トークン
  3. 参照情報合計 = 150,000トークン
  4. 参照情報のうち「新規分」= 150,000 × (1 - 35%) = 97,500トークン
  5. 参照情報のうち「キャッシュ分」= 150,000 × 35% = 52,500トークン
  6. Input = 325 + 97,500 + 1,500(CH オーバーヘッド) = 99,325トークン
  7. Cached Input = 52,500トークン
  8. Output = 500字 × 0.65 = 325トークン
  9. クレジット消費 = [(99,325 × 0.3) + (52,500 × 0.03) + (325 × 1.5)] ÷ 10,000
                  ≒ (29,797.5 + 1,575 + 487.5) ÷ 10,000 ≒ 3.186クレジット (約31.86円)
```

**要点**: 2回目リクエストはキャッシュ効果で「新規情報」の Input単価のみ適用 → クレジット節約

---

#### 🔧 詳細: Input トークン数の組み立て

```
Input トークン数 = プロンプト文字数 + 参照情報(新規分) + 過去ターン + 機能別オーバーヘッド

機能別オーバーヘッド: Chat = 1,500トークン / CLI = 2,500トークン
```

---

#### 📋 文字数 → トークン換算係数

| 文字種 | 換算係数 | 例 |
|---|---|---|
| 日本語 | 1字 ≒ 0.65トークン | 1,000字 ＝ 約650トークン |
| 英語 | 1字 ≒ 0.25トークン | 1,000字 ＝ 約250トークン |

#### 📁 添付ファイル種別ごとの KB → トークン換算係数

| ファイル形式 | 換算係数 (KB単価) | 換算理由 | 計算例 |
|---|---|---|---|
| Markdown / テキスト | 650 | テキストがそのままトークン化 | 10KB ＝ 6,500トークン |
| PDF | 400 | 画像・フォント含む、実テキスト率≒60% | 10KB ＝ 4,000トークン |
| Word (.docx) | 300 | ZIP+XML構造、実テキスト率≒40-50% | 10KB ＝ 3,000トークン |
| PowerPoint (.pptx) | 250 | スライド装飾多、実テキスト率≒35% | 10KB ＝ 2,500トークン |
| Excel (.xlsx) | 200 | セル値のみ実データ、構造情報多、実テキスト率≒30% | 10KB ＝ 2,000トークン |

複数ファイル添付時は、ファイルごとに計算後、単純合計:  
`添付ファイルトークン合計 = Σ(各ファイル size_KB × 該当形式の係数)`

#### 🔄 キャッシュ率の自動計算

利用者は「参照情報の再利用状況」を選択肢から選ぶだけ → 以下のキャッシュ率が自動適用:

| 利用者の選択肢 | キャッシュ率 | 説明 |
|---|---|---|
| 初回依頼 / 参照情報なし | 0% | 参照がないため、キャッシュなし |
| 同じ資料を使う2回目 | 25% | 前回の参照情報が部分的にキャッシュ |
| 同じ資料で2〜3往復 | 35% | キャッシュがより効果的 |
| 同じ資料で4往復以上 | 50% | キャッシュ上限に近い |
| （CLI のみ）同一セッション継続 | +10% | 同セッション内なら追加10% |
| **上限** | **60%** | 技術的上限 |

**計算公式**:  
```
実キャッシュ率 = min(基礎率 + CLI同一セッション補正, 60%)
Cached Input = 参照情報トークン数 × 実キャッシュ率
```

### ② Copilot Code Review の消費クレジット（完全に別ロジック）

**特徴**: Chat/CLI と異なり、モデル単価を使わず、PR の**変更行数（diff行数）**のみで計算します。

#### 基本計算式

```
消費クレジット = 変更行数(diff行数) × 暫定係数(0.05)
```

#### 📝 暫定係数「0.05」の背景

- **実測データなし**: GitHub Copilot code review の消費トークン・クレジットについて、公式の実測データが現時点で公開されていません
- **暫定値**: 0.05クレジット/変更行 は、他のモデル単価と AI 計算量から逆算した仮定値です
- **今後更新**: 実測データが得られ次第、この係数を更新予定です（変数名に `EXPERIMENTAL` 接頭辞を付けています）

#### 🔍 実例計算（シナリオ別）

**【Scenario C】小規模 Code Review PR（10 行変更）**

```
入力:
  - PR の変更行数: 10行

計算:
  消費クレジット = 10行 × 0.05 = 0.5クレジット (約5円)

用途例: バグ修正、1〜2ファイルの小規模変更、コメント修正 など
```

**【Scenario D】中規模 Code Review PR（100 行変更）**

```
入力:
  - PR の変更行数: 100行

計算:
  消費クレジット = 100行 × 0.05 = 5クレジット (約50円)

用途例: 新機能追加、複数ファイルの機能改善 など
```

**【Scenario E】大規模 Code Review PR（500 行変更）**

```
入力:
  - PR の変更行数: 500行

計算:
  消費クレジット = 500行 × 0.05 = 25クレジット (約250円)

用途例: 大規模リファクタリング、複数機能の並行実装 など
```

#### ⚠️ 注意事項

- **diff 行数の定義**: 追加行数 + 削除行数（コンテキスト行は含まない）
- **係数の確定時期**: 2026年実測データ入手後、この説明と `src/rates.js` を同時更新
- **Chat/CLI と異なり**: キャッシュ、オーバーヘッド、複数ファイル形式の考慮はなし（シンプル設計）

## 変数 / 定数 まとめ（クイックリファレンス）

### 📌 利用者が「変えられるもの」（変数・入力値）

| 項目 | 定義 | 例 |
|---|---|---|
| **機能選択** | Chat / CLI / Code Review のいずれか | 「Chat」選択 |
| **使用モデル** | Claude 3.5 Sonnet など | Claude 3.5 Sonnet 選択 |
| **プロンプト文字数** | 質問文の文字数 | 500字入力 |
| **参照情報** | 参照テキスト、コード、添付ファイル(種類・KB) | Word 500KB 添付 |
| **過去会話ターン数** | 前の会話がいくつあるか | 3ターン選択 |
| **想定回答文字数** | AI の回答として予想される文字数 | 1,000字と推定 |
| **再利用状況** | 参照情報の使用頻度 | 「同じ資料で2〜3往復」選択 |
| **CLI同一セッション** | (CLI のみ) 同じセッション内か | チェック ON |
| **変更行数** | (Code Review のみ) PR の diff 行数 | 100行 |

### 🔧 システムが「管理する定数」（固定値・係数）

| 項目 | 定義 | 値 | 管理場所 |
|---|---|---|---|
| **モデル別単価** | Input/Cached Input/Output の $単価 | Claude 3.5 Sonnet: input 0.3円/1K トークン など | `src/rates.js` → `MODEL_RATES` |
| **機能別オーバーヘッド** | Chat/CLI の固定コスト（システムプロンプト等） | Chat: 1,500トークン、CLI: 2,500トークン | `src/rates.js` → `FEATURE_OVERHEAD_TOKENS` |
| **文字→トークン係数** | 日本語・英語の文字数→トークン変換 | 日本語: 0.65、英語: 0.25 | `src/tokenizer.js` |
| **ファイル形式係数** | ファイル形式別の KB→トークン変換 | Markdown: 650、PDF: 400、Word: 300 等 | `src/tokenizer.js` |
| **キャッシュ率テーブル** | 再利用状況ごとのキャッシュ率 | 0%, 25%, 35%, 50%, (CLI+10%), 上限60% | `src/rates.js` → `CACHE_RATES` |
| **Code Review係数** | diff行数→クレジット | 0.05 (暫定・実測前) | `src/rates.js` → `EXPERIMENTAL_CODE_REVIEW_...` |
| **$0.01 = 1クレジット** | 課金単位 | 固定 | 計算式に組み込み済み (`÷ 10,000`) |

### 💡 要点

- **利用者は「何を・どれだけ入力/選択するか」だけを決める** → 表の上半分は変数
- **システム（事務局）が「いくらかかるか」を決める** → 表の下半分は定数
- **単価改定時は `src/rates.js` を更新するだけ** → 全計算に反映される設計
- **テスト時は `src/tokenizer.js` と `src/calculator.js` の関数が正確かを確認** → Jest で自動テスト推奨
