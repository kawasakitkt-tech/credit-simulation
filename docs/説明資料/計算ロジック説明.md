# GitHub Copilot クレジット消費量計算ツール 計算ロジック説明

**対象読者**: 全社員向け（承認プロセス、クレジット管理、個人の事前見積もり用）  
**本ツール目的**: 1リクエストあたりのクレジット消費量を**事前に見積もる**スタンドアロン Web ツール

GitHub Copilot は 2026/6/1 に premium request 方式の課金を廃止し、トークンベースの従量課金（GitHub AI Credits, 1 credit = $0.01）に移行しました。本ツールは **Ask モード / Plan モード / Agent モード** の3つを対象とします。

---

## 用語定義（5分で分かる基礎）

| 用語 | 説明 | 例 |
|---|---|---|
| **トークン** | AI モデルが文字列を処理する際の最小単位。日本語は1文字≒0.65トークン、英語は1文字≒0.25トークン | 「こんにちは」(5字) ≒ 3.25 トークン |
| **Input トークン** | AI に送信する情報（プロンプト＋参照ファイル）のトークン数 | 500 字プロンプト＋10KB資料 ＝ 約5,000トークン |
| **Output トークン** | AI が返す回答のトークン数 | 1,000字回答 ≒ 650 トークン |
| **Cached Input** | 前回リクエストで使用した参照情報を AI キャッシュから再利用した場合のトークン数（単価が通常の1/10） | 同じ資料を2回目に使用 → キャッシュ効果で安価 |
| **Cache Write** | エージェントループ（Plan/Agent）で新規コンテキストをキャッシュに書き込む分。Anthropic 系は専用単価、他は input 単価で課金 | 反復1回目に読み込んだ参照情報をキャッシュへ書き込む |
| **クレジット** | GitHub Copilot の課金単位。1クレジット = $0.01 | 5,000トークン = 0.5〜1.5クレジット（モデルにより異なる） |

---

## 全体構造(2つの計算式)

本ツールは「①Ask」と「②Plan/Agent」で計算式が異なります。Ask は1回のモデル呼び出しで完結しますが、Plan/Agent は内部でモデル呼び出しが自律的に複数回起こる前提の**反復シミュレーション**で近似します。

---

### ① Ask モードの消費クレジット（基本計算式）

```
消費クレジット = [(Input × Input単価) + (Cached Input × Cached単価) + (Output × Output単価)] ÷ 10,000
```

**登場人物:**
- **Input トークン数**: プロンプト + 参照情報（再利用されない分） + 過去ターンの履歴 + オーバーヘッド
- **Cached Input トークン数**: 参照情報・履歴のうち「キャッシュから再利用できた分」
- **Output トークン数**: AI の回答文
- **モデル別単価**: Input/Cached Input/Output それぞれで異なる固定値（例: Claude Sonnet 5 は Input 2.00 USD/1M トークン）

**「何回目のやり取りか」がすべてを決める**: Ask では、利用者が入力する「今回のリクエストが同一スレッドの何回目のやり取りか（ターン数 T）」が、①過去ターンの履歴トークン量と②キャッシュ率の両方を自動的に決定します。利用者がキャッシュ率を直接入力する必要はありません。

- 履歴トークン: `(T-1) × 平均ターンサイズ`（平均ターンサイズ＝ユーザー500文字＋アシスタント1,000文字）
- キャッシュ率: T=1→0%、T=2→25%、T=3〜4→35%、T≥5→50%（参照情報＋履歴の合計に適用）

#### 📊 実例ベース計算（シナリオ別）

**【Scenario A】1回限りの Ask リクエスト（参照なし・初回）**

```
入力内容:
  - プロンプト文字数: 500字
  - 参照ファイル・テキスト: なし
  - 何回目のやり取りか: 1（初回）
  - 想定回答文字数: 300字

計算ステップ:
  1. プロンプトのトークン数 = 500字 × 0.65 = 325トークン
  2. 参照情報トークン = 0（参照なし）
  3. Ask 固定オーバーヘッド = 1,500トークン
  4. T=1 のため履歴トークン = 0、キャッシュ率 = 0%
  5. Input合計 = 325 + 0 + 1,500 = 1,825トークン
  6. Cached Input = 0
  7. Output = 300字 × 0.65 = 195トークン
  8. クレジット消費 = [(1,825 × Input単価) + (0 × Cached単価) + (195 × Output単価)] ÷ 10,000
```

**【Scenario B】同じ Word 資料について 2 回目の Ask（T=2）**

```
入力内容:
  - プロンプト文字数: 500字（1回目と同じ資料について質問）
  - Word 添付ファイル: 500KB
  - 何回目のやり取りか: 2
  - 想定回答文字数: 500字

計算ステップ:
  1. プロンプト = 500字 × 0.65 = 325トークン
  2. Word資料のトークン = 500KB × 300(KB単価) = 150,000トークン
  3. T=2 のため履歴トークン = 1ターン分（ユーザー500字＋アシスタント1,000字の概算）が加算される
  4. T=2 のためキャッシュ率 = 25%（参照情報＋履歴の合計に適用）
  5. キャッシュ対象合計 = 150,000 + 履歴トークン
  6. Cached Input = キャッシュ対象合計 × 25%
  7. 新規分（Input側） = キャッシュ対象合計 × (1 - 25%)
  8. Input = プロンプト325 + 新規分 + Ask オーバーヘッド1,500
  9. Output = 500字 × 0.65 = 325トークン
  10. クレジット消費 = [(Input × Input単価) + (Cached Input × Cached単価) + (325 × Output単価)] ÷ 10,000
```

**要点**: ターン数が進むほど履歴分の Input は増えるが、同じ参照情報・履歴はキャッシュ割引が効くため、新規情報の単価のみが上乗せされる。

---

#### 🔧 詳細: Input トークン数の組み立て

```
Input トークン数 = プロンプト文字数 + 参照情報(新規分) + 履歴(新規分) + モード別オーバーヘッド

モード別オーバーヘッド: Ask = 1,500トークン
```

---

#### 📋 文字数 → トークン換算係数

| 文字種 | 換算係数 | 例 |
|---|---|---|
| 日本語 | 1字 ≒ 0.65トークン | 1,000字 ＝ 約650トークン |
| 英語 | 1字 ≒ 0.25トークン | 1,000字 ＝ 約250トークン |

#### 📁 添付ファイル種別ごとの KB → トークン換算係数

| ファイル形式 | 換算係数 (KB単価) | 換算理由 | 計算例 |
|---|---|---|---|
| Markdown / テキスト | 650 | テキストがそのままトークン化 | 10KB ＝ 6,500トークン |
| PDF | 400 | 画像・フォント含む、実テキスト率≒60% | 10KB ＝ 4,000トークン |
| Word (.docx) | 300 | ZIP+XML構造、実テキスト率≒40-50% | 10KB ＝ 3,000トークン |
| PowerPoint (.pptx) | 250 | スライド装飾多、実テキスト率≒35% | 10KB ＝ 2,500トークン |
| Excel (.xlsx) | 200 | セル値のみ実データ、構造情報多、実テキスト率≒30% | 10KB ＝ 2,000トークン |
| ソースコード / フォルダ | 300 | ASCII主体のため言語共通（要確認・暫定値） | 10KB ＝ 3,000トークン |

複数ファイル添付時は、ファイルごとに計算後、単純合計:  
`添付ファイルトークン合計 = Σ(各ファイル size_KB × 該当形式の係数)`

#### 🔄 「何回目のやり取りか」→ キャッシュ率の自動計算

利用者は「何回目のやり取りか」を入力するだけ → 以下のキャッシュ率が自動適用:

| 何回目のやり取りか (T) | キャッシュ率 | 説明 |
|---|---|---|
| T = 1（初回） | 0% | 参照・履歴とも新規のためキャッシュなし |
| T = 2 | 25% | 前回の参照情報・履歴が部分的にキャッシュ |
| T = 3〜4 | 35% | キャッシュがより効果的 |
| T ≥ 5 | 50% | キャッシュ上限に近い |

**計算公式**:  
```
Cached Input = (参照情報トークン + 履歴トークン) × キャッシュ率
```

### ② Plan / Agent モードの消費クレジット（反復シミュレーション）

**特徴**: Ask と異なり、Plan/Agent は「モデルが自律的にコードベースを探索・編集しながら複数回モデル呼び出しを行うループ」を反復シミュレーションの閉形式で近似します。

#### 基本計算式（1回目に初期コンテキストをキャッシュ書き込み、以降は蓄積分を読み込みながら増分を書き込む）

```
cacheWriteTokens  = C0 + (N-1)・G
cachedInputTokens = (N-1)・C0 + G・(N-1)(N-2)/2
outputTokens      = N・O + F

消費クレジット = [(Cache Write × Cache Write単価) + (Cached Input × Cached単価) + (Output × Output単価)] ÷ 10,000
```

**登場人物:**
- **C0（初回送信コンテキスト）**: モードオーバーヘッド（Plan=4,000 / Agent=6,000トークン）＋プロンプト＋参照情報
- **N（反復回数）**: モデル呼び出し回数（ツール呼び出しラウンド数）。最小1回にクランプ
- **G（増分トークン）**: 1反復ごとに追加されるコンテキスト（ツール実行結果＋前回出力）
- **O（反復あたり出力）**: 1反復あたりの出力トークン（思考・ツール呼び出し指示）
- **F（最終成果物）**: 最終回に追加される出力（Plan=計画書、Agent=コード＋説明）
- **サブエージェント**: 1体ごとに同エンジンの小型ループ（既定 N=6, G=3,000, O=800）として計算し合算

Cache Write 単価を持たないモデル（OpenAI / Google / GitHub / Microsoft 系）は、Cache Write トークンを通常の Input 単価で課金します。Anthropic 系モデルのみ専用の Cache Write 単価が設定されています。

#### 🔍 タスク規模プリセット（暫定値・要較正）

利用者は「タスク規模」を選ぶだけで、以下のプリセットが自動適用されます（詳細調整で個別に上書き可）。

| モード | 規模 | N（反復回数） | G（増分/反復） | O（出力/反復） | F（最終出力） | サブエージェント数 |
|---|---|---|---|---|---|---|
| Plan | 軽微 | 4 | 3,000 | 300 | 2,000 | 0 |
| Plan | 中規模 | 8 | 4,000 | 400 | 4,000 | 0 |
| Plan | 大規模 | 14 | 5,000 | 500 | 8,000 | 2 |
| Agent | 軽微 | 6 | 3,000 | 800 | 1,000 | 0 |
| Agent | 中規模 | 15 | 4,000 | 1,000 | 1,500 | 1 |
| Agent | 大規模 | 30 | 5,000 | 1,200 | 2,000 | 3 |

**用途の目安**:
- 軽微: 数ファイルの小修正・バグ修正
- 中規模: 一機能の実装
- 大規模: 横断的な変更・大きな新機能（サブエージェントを併用）

#### 📝 プリセット係数の背景

- **実測データなし**: GitHub Copilot の Plan/Agent モードにおける反復回数・増分トークン等について、公式の実測データは現時点で公開されていません
- **暫定値**: 上記プリセットは、エージェントループの一般的な挙動から逆算した仮定値です
- **今後更新**: 実測データが得られ次第、この係数を更新予定です（変数名に `EXPERIMENTAL` 接頭辞を付けています）

#### 🔍 実例計算（シナリオ別・Agent モード）

**【Scenario C】Agent・軽微タスク（プリセットそのまま）**

```
入力:
  - モード: Agent、タスク規模: 軽微（N=6, G=3,000, O=800, F=1,000, サブエージェント0）
  - プロンプト＋参照情報＋オーバーヘッド(6,000) = C0（例: 10,000トークン）

計算:
  cacheWriteTokens  = 10,000 + 5×3,000 = 25,000
  cachedInputTokens = 5×10,000 + 3,000×5×4/2 = 50,000 + 30,000 = 80,000
  outputTokens      = 6×800 + 1,000 = 5,800

用途例: 数ファイルの小規模なコード修正・バグ修正
```

**【Scenario D】Agent・中規模タスク（サブエージェント1体を含む）**

```
入力:
  - モード: Agent、タスク規模: 中規模（N=15, G=4,000, O=1,000, F=1,500, サブエージェント1）
  - C0（例: 10,000トークン）
  - サブエージェント1体分（既定: N=6, C0=3,000+参照トークン×0.5, G=3,000, O=800, F=0）を加算合算

計算（メインループのみ）:
  cacheWriteTokens  = 10,000 + 14×4,000 = 66,000
  cachedInputTokens = 14×10,000 + 4,000×14×13/2 = 140,000 + 364,000 = 504,000
  outputTokens      = 15×1,000 + 1,500 = 16,500

  → これにサブエージェント1体分のトークンを加算した値が最終合計になる

用途例: 一機能の実装（テスト追加含む）
```

**【Scenario E】Agent・大規模タスク（サブエージェント3体を含む）**

```
入力:
  - モード: Agent、タスク規模: 大規模（N=30, G=5,000, O=1,200, F=2,000, サブエージェント3）
  - C0（例: 15,000トークン）

計算（メインループのみ）:
  cacheWriteTokens  = 15,000 + 29×5,000 = 160,000
  cachedInputTokens = 29×15,000 + 5,000×29×28/2 = 435,000 + 2,030,000 = 2,465,000
  outputTokens      = 30×1,200 + 2,000 = 38,000

  → サブエージェント3体分のトークンをそれぞれ加算した値が最終合計になる

用途例: 横断的な大規模リファクタリング、複数機能の並行実装
```

#### ⚠️ 注意事項

- **反復回数・増分・出力の定義**: プリセットは目安であり、実際のタスク内容によって大きく変動する
- **係数の確定時期**: 実測データ入手後、この説明と `src/rates.js` を同時更新
- **Ask と異なり**: 「何回目のやり取りか」の概念はなく、タスク規模プリセット（＋詳細調整）が反復回数を決める

## 変数 / 定数 まとめ（クイックリファレンス）

### 📌 利用者が「変えられるもの」（変数・入力値）

| 項目 | 定義 | 例 |
|---|---|---|
| **モード選択** | Ask / Plan / Agent のいずれか | 「Ask」選択 |
| **使用モデル** | Claude Sonnet 5 など | Claude Sonnet 5 選択 |
| **プロンプト文字数** | 質問文の文字数 | 500字入力 |
| **参照情報** | 参照テキスト、コード、添付ファイル(種類・KB) | Word 500KB 添付 |
| **何回目のやり取りか**（Ask のみ） | 同一スレッドの何ターン目か | 3ターン目 |
| **想定回答文字数**（Ask のみ） | AI の回答として予想される文字数 | 1,000字と推定 |
| **タスク規模**（Plan/Agent のみ） | 軽微 / 中規模 / 大規模 | 「中規模」選択 |

### 🔧 システムが「管理する定数」（固定値・係数）

| 項目 | 定義 | 値 | 管理場所 |
|---|---|---|---|
| **モデル別単価** | Input/Cached Input/Cache Write/Output の $単価 | モデルにより異なる（例: Claude Sonnet 5 は Input 2.00 USD/1Mトークン） | `src/rates.js` → `MODEL_RATES` |
| **モード別オーバーヘッド** | Ask/Plan/Agent の固定コスト（システムプロンプト等） | Ask: 1,500 / Plan: 4,000 / Agent: 6,000トークン | `src/rates.js` → `FEATURE_OVERHEAD_TOKENS` |
| **文字→トークン係数** | 日本語・英語の文字数→トークン変換 | 日本語: 0.65、英語: 0.25 | `src/tokenizer.js` |
| **ファイル形式係数** | ファイル形式別の KB→トークン変換 | Markdown: 650、PDF: 400、Word: 300、ソースコード: 300 等 | `src/tokenizer.js` |
| **Ask キャッシュ率テーブル** | 「何回目のやり取りか」ごとのキャッシュ率 | T=1→0%、T=2→25%、T=3〜4→35%、T≥5→50% | `src/rates.js` → `ASK_TURN_CACHE_RATIOS` |
| **Plan/Agent タスク規模プリセット** | 反復回数・増分・出力・サブエージェント数 | 軽微/中規模/大規模ごとに定義（暫定・実測前） | `src/rates.js` → `EXPERIMENTAL_AGENTIC_PRESETS` |
| **サブエージェント既定値** | サブエージェント1体分の固定小型ループ | N=6, G=3,000, O=800 等（暫定・実測前） | `src/rates.js` → `EXPERIMENTAL_SUBAGENT_DEFAULTS` |
| **$0.01 = 1クレジット** | 課金単位 | 固定 | 計算式に組み込み済み (`÷ 10,000`) |

### 💡 要点

- **利用者は「何を・どれだけ入力/選択するか」だけを決める** → 表の上半分は変数
- **システム（事務局）が「いくらかかるか」を決める** → 表の下半分は定数
- **単価改定時は `src/rates.js` を更新するだけ** → 全計算に反映される設計
- **`EXPERIMENTAL_` の付く係数は実測前の暫定値** → 実測データ取得後に較正予定
- **テスト時は `src/tokenizer.js` と `src/calculator.js` の関数が正確かを確認** → Jest で自動テスト推奨
