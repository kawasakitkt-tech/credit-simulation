# GitHub Copilot クレジット消費量計算ツール 計算ロジック説明

対象読者: 役員・事務局向け（承認プロセスでの説明用）

## 全体構造(2つの計算式)

本ツールは「①Chat/CLI」と「②Code Review」で計算式が完全に別物です。

---

### ① Chat / CLI の消費クレジット

```
消費クレジット = Σ (トークン数 × モデル別単価) ÷ 10,000
```

対象は **Input / Cached Input(再利用分) / Output** の3種類のトークン(Cache Writeは未使用=常に0)。トークン種別ごとに単価が異なるため、実質は3つの掛け算の合計です。

```
消費クレジット
 = (Inputトークン数 × Input単価)
 + (Cached Inputトークン数 × Cached Input単価)
 + (Outputトークン数 × Output単価)
   ÷ 10,000   ← $0.01/クレジット換算込みの定数
```

**単価(Input/Cached Input/Output)はモデルごとに固定値** (`src/rates.js` の `MODEL_RATES`)。利用者が選んだモデルによってこの単価が切り替わるだけで、利用者が単価を入力することはありません。

#### Inputトークン数の組み立て

```
Inputトークン数
 = (プロンプト文字数 → トークン換算)          ← 変数(入力文言)
 + (参照情報トークンのうち「再利用されない分」)  ← 変数(参照テキスト/添付ファイル量) × (1−キャッシュ率)
 + (過去会話ターン数 → トークン換算)          ← 変数(ターン数)
 + (機能ごとの固定オーバーヘッド)             ← 定数(Chat=1,500 / CLI=2,500 トークン)
```

「文字数 → トークン換算」の係数(1文字あたり何トークンか)も**定数**です(日本語0.65、英語0.25。添付ファイルは種別ごとにKBあたりの係数がさらに定数として存在)。

#### Cached Inputトークン数(＝再利用分)

```
Cached Inputトークン数 = 参照情報トークン数[変数] × キャッシュ率
```

キャッシュ率は利用者が%を直接入力するのではなく、**「参照情報の再利用状況」という選択肢[変数]から自動算出**します。

```
キャッシュ率 = min( 基礎率 + CLI同一セッション補正 , 上限60% )
```

| 選択肢(変数) | 基礎率(定数) |
|---|---|
| 初回依頼 / 参照情報なし | 0% |
| 同じ資料を使う2回目 | 25% |
| 同じ資料で2〜3往復 | 35% |
| 同じ資料で4往復以上 | 50% |
| (CLIのみ)同一セッション継続チェック | +10%(定数) |
| 上限 | 60%(定数) |

※参照情報トークンが0の場合は選択に関わらず必ず0%。

#### 添付ファイルのトークン換算(ファイル種別で係数が異なる)

添付ファイルは「サイズ(KB)」を入力するだけですが、そこから何トークンとみなすかの換算係数はファイル種別ごとに固定値として異なります。

```
添付ファイル1件あたりのトークン数 = ファイルサイズ(KB)[変数] × ファイル種別ごとの換算係数(定数)
```

| ファイル形式 | 換算係数(KBあたりトークン数・定数) | 係数が低くなる理由 |
|---|---|---|
| Markdown / テキスト | 650 | テキストがほぼそのままトークンになるため最大 |
| PDF | 400 | 画像・フォントデータを含み実テキスト率は約60% |
| Word(.docx) | 300 | ZIP+XML構造のため実テキスト率は約40〜50% |
| PowerPoint(.pptx) | 250 | スライド装飾要素が多く実テキスト率は約35% |
| Excel(.xlsx) | 200 | セル値以外の構造情報が多く実テキスト率は約30% |

複数ファイルを添付した場合は、ファイルごとにこの式で計算してから単純合計します。

```
添付ファイルトークン数(合計) = Σ (各ファイルのサイズKB[変数] × 該当ファイル種別の係数[定数])
```

この合計は、本文欄に貼り付けた「参照コード・設計書テキスト」のトークン数と合算されて「参照情報トークン数」となり、上記のInputトークン計算とキャッシュ率適用(Cached Inputへの振り分け)の対象になります。ファイル形式が変わればトークン数の見積もりが変わり、結果として消費クレジットも変わる、という点がポイントです。

※ 係数は日本語コンテンツを想定した値です。内部的には英語用の係数(md=250, pdf=180, docx=130, pptx=110, xlsx=100)も用意されていますが、現行UIには言語を切り替える項目が無く、常に日本語係数を使用する仕様になっています。

#### Outputトークン数

```
Outputトークン数 = 想定回答文字数[変数] → トークン換算(係数は定数)
```

---

### ② Copilot code review の消費クレジット

Chat/CLIとは全く別ロジックで、モデル単価は使いません。

```
消費クレジット = 変更行数(diff行数)[変数] × 暫定係数(定数・実測前の仮値0.05)
```

この係数は実測データがまだ無いため暫定値扱いで、実測後に更新予定です。

---

## 変数 / 定数 まとめ

| 区分 | 項目 | 誰が決めるか |
|---|---|---|
| **変数(利用者入力)** | 機能(Chat/CLI/Code review)、使用モデル、プロンプト文字数、参照テキスト量、添付ファイルの種類・サイズ(KB)、過去会話ターン数、想定回答文字数、参照情報の再利用状況、CLI同一セッション継続有無、変更行数 | 利用者が画面で選択・入力 |
| **定数(システム管理)** | モデル別単価(Input/Cached Input/Output)、機能別オーバーヘッド(Chat1,500/CLI2,500)、文字数→トークン換算係数、**ファイル種別ごとのKB→トークン換算係数(md/pdf/docx/pptx/xlsx で異なる)**、キャッシュ率テーブル(0/25/35/50%、CLI+10%、上限60%)、code review暫定係数(0.05)、$0.01=1クレジットの換算率 | 事務局が `src/rates.js`(単価等)・`src/tokenizer.js`(換算係数)で一元管理・更新 |

**要点:** 利用者が変えられるのは「何を・どれだけ入力/選択するか」だけで、単価やキャッシュ率テーブルなどの“係数”はすべてシステム側の定数として固定されています。単価改定時は `src/rates.js` の数値を書き換えるだけで全計算に反映される設計です。
